    Introduction

    Introduction

    Dans le monde actuel, la rapidité et la flexibilité sont devenues essentielles. Les entreprises n’achètent plus forcément de serveurs physiques pour exécuter leurs applications : elles louent de la puissance de calcul dans le cloud. C’est exactement ce que propose Amazon EC2 (Elastic Compute Cloud).

    EC2 permet de lancer en quelques clics des instances, c’est-à-dire des serveurs virtuels configurés sur mesure. Le choix est large : système d’exploitation Linux ou Windows, processeurs Intel, AMD ou AWS Graviton pour des performances optimisées, et de nombreuses tailles d’instances adaptées aussi bien aux tests qu’aux environnements critiques de production.

    La tarification est simple et attractive : on paie uniquement ce qu’on consomme (pay-as-you-go). Mieux encore, les nouveaux utilisateurs bénéficient d’un forfait gratuit de 750 heures par mois pendant 12 mois.

    Avec EC2, on peut héberger un site web, déployer un ERP ou une base de données, créer un environnement de développement ou encore traiter de grandes quantités de données. Tout cela, sans se soucier de l’infrastructure physique : AWS s’en occupe.

    Ce rapport a pour objectif de guider le lecteur dans la découverte d’Amazon EC2, depuis le lancement d’une première instance jusqu’à la mise en place de solutions avancées comme l’auto scaling et le load balancing.

    1. Caractéristiques principales d’Amazon EC2

    Amazon EC2 repose sur un ensemble d’éléments qui définissent l’identité, la configuration et la sécurité de chaque instance. Ces caractéristiques sont essentielles à comprendre avant de lancer et gérer ses premiers serveurs virtuels.

    1.1 Balises (Tags)

    Les balises sont des métadonnées qu’on associe à une instance EC2. Elles prennent la forme de paires clé–valeur (par exemple : Name = SRV01). Les balises facilitent l’organisation, le tri et l’automatisation. Elles servent également au suivi des coûts et permettent de savoir à quel environnement (production, test, développement) ou à quel propriétaire une ressource appartient.

    1.2 AMI (Amazon Machine Image)

    Une AMI est un modèle préconfiguré qui permet de choisir le système d’exploitation et, dans certains cas, des logiciels déjà installés. C’est l’équivalent d’une image ISO dans un environnement classique. Grâce aux AMI, il est possible de déployer rapidement une instance Linux, Windows ou même une image personnalisée créée par un fournisseur tiers ou par l’utilisateur lui-même.

    1.3 Types d’instances

    Le type d’instance détermine la configuration matérielle virtuelle : nombre de vCPU, mémoire vive, performance réseau, capacité de stockage. AWS propose une large gamme, allant des micro-instances pour des tests aux machines puissantes destinées aux applications critiques. Le choix du type d’instance se fait en fonction des besoins en calcul, mémoire et réseau.

    1.4 Key Pair (Paires de clés)

    Les paires de clés sont un mécanisme d’authentification sécurisé. Elles se composent d’une clé publique stockée sur l’instance et d’une clé privée téléchargée par l’utilisateur. Cette clé privée permet d’ouvrir une connexion SSH (pour Linux) ou de déchiffrer le mot de passe administrateur (pour Windows). Bien que la création d’une Key Pair soit optionnelle, elle reste la méthode la plus courante pour accéder en toute sécurité à une instance.

    1.5 Réseau et Sous-réseaux (Subnet)

    Chaque instance EC2 est déployée à l’intérieur d’un VPC (Virtual Private Cloud). Elle est rattachée à un sous-réseau, associé à une zone de disponibilité précise. Une instance peut recevoir une adresse IP privée (pour communiquer avec d’autres instances dans le VPC) et, de manière optionnelle, une adresse IP publique (pour être accessible depuis Internet).

    1.6 Groupes de sécurité

    Un groupe de sécurité agit comme un pare-feu virtuel. Il contrôle le trafic entrant (inbound) et sortant (outbound) des instances. Les règles sont définies par protocole, par port et par adresse source ou destination. Contrairement aux pare-feux classiques, les groupes de sécurité appliquent un deny implicite : tout ce qui n’est pas explicitement autorisé est automatiquement bloqué.

    1.7 Volumes EBS (Elastic Block Store)

    Les volumes EBS sont des disques virtuels en réseau attachés aux instances EC2. Le volume principal (root volume) contient le système d’exploitation, mais il est possible d’ajouter des volumes supplémentaires pour stocker des données. Les volumes EBS sont persistants : les données sont conservées même si l’instance est arrêtée, sauf en cas de suppression explicite du volume.

    LAB : Lancer une instance EC2

    Avant de démarrer, il est recommandé de se rendre sur le site CloudPing afin de mesurer la latence entre votre poste et les différentes régions AWS. Cela permet de choisir la région la plus proche et donc la plus performante.

    Ensuite, dans la console de gestion AWS, utilisez la barre de recherche pour taper “EC2” puis cliquez sur le service EC2.

    On clique sur ‘ lancer une instance ‘

    Lorsque l’on clique sur “Lancer une instance”, une page de configuration s’ouvre. Elle permet de renseigner toutes les informations nécessaires au déploiement de l’instance EC2.

    La première étape consiste à définir des tags. On peut, par exemple, ajouter un tag Name = SRV01 pour identifier facilement le serveur.On peut également créer un tag supplémentaire, par exemple Groupe = Production, afin d’indiquer que cette instance appartient à l’environnement de production

    Ensuite, il faut choisir une AMI (Amazon Machine Image). Dans notre cas, nous sélectionnons Amazon Linux 2023, la distribution Linux officielle fournie et maintenue par AWS.

    Nous sélectionnons ensuite une image correspondant à l’architecture système souhaitée (par exemple x86_64 ou ARM64), selon le type de processeur et les besoins de l’application.

    Dans notre cas, nous choisissons une instance de type t3.micro, disposant de 2 vCPU et de 1 Go de mémoire RAM. Ce type d’instance fait partie des ressources éligibles à l’offre gratuite AWS pour les nouveaux comptes.

    Pour accéder à l’instance, il est nécessaire de créer une Key Pair (paire de clés).On clique sur “Créer une paire de clés”, puis AWS génère automatiquement une clé publique (enregistrée sur l’instance) et une clé privée (téléchargée localement). Cette clé privée servira ensuite à établir une connexion sécurisée via SSH (Linux) ou RDP (Windows).

    On attribue un nom à la Key Pair, par exemple key-pair-SRV01, puis on choisit le type de clé (RSA) ainsi que le format du fichier de la clé privée (PEM ou PPK selon l’outil de connexion utilisé

    Après avoir cliqué sur “Créer”, AWS génère la paire de clés et télécharge automatiquement la clé privée sur la machine locale. Il est important de la conserver en lieu sûr, car elle sera indispensable pour se connecter à l’instance.

    Pour la configuration réseau, on sélectionne le VPC par défaut, puis un sous-réseau. Chaque sous-réseau est toujours associé à une zone de disponibilité.On active également l’option “Attribuer une adresse IP publique”, ce qui permet à l’instance EC2 d’obtenir une adresse accessible depuis Internet.

    On crée ensuite un groupe de sécurité nommé prod-firewall.Par défaut, AWS y ajoute une règle entrante autorisant les connexions SSH (port 22) depuis n’importe quelle adresse IP. Cette configuration est pratique pour des tests rapides, mais il est recommandé, pour plus de sécurité, de restreindre la source aux adresses IP autorisées.

    Pour la partie stockage, un seul volume est créé par défaut : le root volume, qui contient le système d’exploitation.On peut toutefois ajouter un volume EBS secondaire afin de stocker les données séparément du système.

    Dans la section Détails avancés, plusieurs options supplémentaires sont disponibles (configuration DNS, rôles IAM, scripts de démarrage, etc.). Pour ce déploiement, elles ne seront pas modifiées.

    Un récapitulatif des configurations apparaît dans le panneau de gauche. Après vérification, il suffit de cliquer sur “Lancer l’instance” pour démarrer le déploiement de la machine EC2.

    L’instance passe alors en phase de lancement. AWS initialise les ressources nécessaires avant qu’elle ne devienne disponible.

    On clique ensuite sur “Afficher toutes les instances”, puis on sélectionne l’instance créée afin d’afficher ses informations. Avant toute utilisation, il faut patienter jusqu’à la fin des contrôles d’état effectués automatiquement par AWS.

    Dans la section Détails, l’instance dispose d’une adresse IP privée, associée à un nom DNS interne, ainsi que d’une adresse IP publique pour l’accès depuis Internet.

    Dans l’onglet Surveillance, AWS affiche différentes métriques liées à l’instance, comme l’utilisation du CPU, le trafic réseau ou encore les opérations de lecture/écriture sur les volumes EBS.

    Dans l’onglet Groupe de sécurité, on retrouve le pare-feu virtuel associé à l’instance.

    Par défaut, AWS autorise tout le trafic sortant vers n’importe quelle destination.

    On y retrouve également la règle entrante configurée précédemment (par exemple, SSH ou HTTP selon le besoin).

    Dans l’onglet Mise en réseau, figurent les principales informations liées à la connectivité : l’adresse IP publique et privée de l’instance, l’ID du VPC auquel elle est rattachée, ainsi que le sous-réseau associé.

    Dans l’onglet Stockage, apparaissent les volumes EBS associés à l’instance. On retrouve ici le volume racine (root) contenant le système d’exploitation, ainsi que le volume additionnel ajouté pour le stockage des données.

    En bas de la console, des graphiques de surveillance permettent de visualiser en temps réel l’état de l’instance : utilisation CPU, activité réseau et performances du stockage.

    Dans l’onglet Tags, on retrouve l’ensemble des balises associées à l’instance EC2, comme le nom, l’environnement ou le groupe d’appartenance.

    2.2 Connexion à une instance (EC2 Instance Connect, SSH, RDP)

    Connexion à l’instance EC2

    Pour se connecter, il faut sélectionner l’instance dans la console AWS, puis cliquer sur “Se connecter”.

    Deux méthodes de connexion sont disponibles : EC2 Instance Connect, accessible directement depuis la console AWS via un navigateur web, et le client SSH, à utiliser depuis un terminal local.

    Connexion via EC2 Instance Connect

    EC2 Instance Connect permet de se connecter à l’instance directement depuis la console AWS, sans configuration supplémentaire.AWS ouvre une interface web qui donne accès à un terminal CLI de l’instance.Il suffit de cliquer sur “Se connecter” pour établir la session et interagir avec la machine en ligne de commande.

    Une fois sur “Se connecter”, AWS ouvre un nouvel onglet et lance un terminal CLI permettant d’interagir directement avec l’instance EC2.Par exemple, la commande Linux whoami permet de vérifier l’utilisateur actuellement connecté.

    Connexion via un client SSH

    L’autre méthode consiste à utiliser un client SSH, comme PuTTY sous Windows ou la commande ssh sous Linux/Mac.AWS fournit dans la console toutes les étapes nécessaires : préciser l’adresse publique de l’instance, le chemin vers la clé privée téléchargée, puis établir la connexion sécurisée.

    En suivant les étapes indiquées par AWS, la connexion via SSH s’établit sans difficulté et l’accès à l’instance est fonctionnel.

    2.3 Gestion des tags

    Les Tags

    Les tags sont des métadonnées que l’on associe aux ressources AWS, dont les instances EC2. Ils jouent plusieurs rôles :

    Faciliter le classement et le tri des ressources,

    Permettre l’automatisation de certaines tâches,

    Assurer le suivi des coûts en identifiant les ressources par service, projet ou environnement,

    Améliorer l’organisation générale d’une infrastructure.

    Un tag est toujours défini sous la forme d’une paire clé–valeur.Exemple : Name = SRV01, où la clé est Name et la valeur SRV01.

    Le tag le plus courant est Name, utilisé pour nommer et reconnaître rapidement une instance. Mais il est possible de créer des tags personnalisés, comme Environnement = Production ou Département = IT.

    À noter : les tags ne sont pas limités aux instances EC2. Ils peuvent être appliqués à de nombreuses ressources AWS (volumes EBS, groupes de sécurité, VPC, etc.), ce qui facilite la gestion globale du cloud.

    2.4 Utilisation des User Data scripts

    Les User Data (données utilisateur) permettent de transmettre des scripts exécutés automatiquement lors du premier démarrage d’une instance EC2.Cette option est disponible dans la section Détails avancés au moment de la configuration.

    Ils sont particulièrement utiles pour automatiser certaines tâches, comme :

    L’installation des mises à jour système,

    L’installation de logiciels,

    La configuration initiale de l’environnement de travail.

    Par exemple, il est possible de lancer une instance EC2 qui installe et configure directement un serveur web grâce à un simple script User Data.

    LAB : Script User Data

    Configurons et démarrons une nouvelle instance EC2 en y intégrant un script User Data afin d’automatiser son déploiement.

    Dans la section Données utilisateur (User Data), il suffit de coller un script qui sera exécuté automatiquement au démarrage de l’instance.Ce script permettra, par exemple, d’installer un serveur web et d’héberger une page spécifique dès la mise en service.

    Une fois l’instance EC2 démarrée, on copie le nom DNS public de la machine puis on le saisit dans un navigateur web. Cela permet de vérifier que le serveur web a bien été installé et que la page est accessible.

    À ce stade, l’instance fonctionne correctement : le serveur web est opérationnel et la page est bien hébergée dans le cloud AWS. Tout cela a été automatisé grâce au script User Data exécuté lors du démarrage

    Amazon Machine Images (AMI)3.1 Définition et caractéristiques

    Les AMI (Amazon Machine Image) sont des modèles préconfigurés utilisés pour lancer des instances EC2.Elles contiennent un système d’exploitation (Linux, Windows, etc.) et, dans certains cas, des applications ou configurations supplémentaires.

    Par exemple, lorsqu’on lance une instance EC2 avec Windows, cela signifie que l’on utilise une AMI Windows.De la même manière, certains éditeurs de logiciels proposent leurs produits sous forme d’AMI (par exemple un antivirus ou un ERP préinstallé).

    Les AMI peuvent provenir de différentes sources :

    AWS : images officielles maintenues par Amazon,

    la communauté : partagées par d’autres utilisateurs AWS,

    des fournisseurs tiers : éditeurs de logiciels qui packagent et commercialisent leurs solutions via des AMI.

    3.2 Différents types d’AMI

    Les AMI peuvent provenir de plusieurs sources :

    AMI fournies par AWS :Ce sont des images officielles, maintenues et mises à jour par Amazon (par exemple Amazon Linux, Windows Server, ou encore Ubuntu). Elles garantissent stabilité et sécurité.

    AMI de la communauté :Partagées par d’autres utilisateurs AWS, elles offrent une grande variété de systèmes et configurations. Toutefois, leur fiabilité et leur niveau de sécurité peuvent varier, car elles ne sont pas gérées directement par AWS.

    AMI de fournisseurs tiers :Proposées par des éditeurs de logiciels sur l’AWS Marketplace, elles incluent souvent des applications prêtes à l’emploi (exemple : bases de données, ERP, outils de sécurité). Elles peuvent être gratuites ou payantes, selon la licence.

    3.3 Création et utilisation d’AMI personnalisées (LAB)

    À partir d’une seule AMI, il est possible de lancer plusieurs instances EC2 identiques. Cependant, ces instances ne partagent aucun lien direct : elles fonctionnent de manière totalement indépendante.

    Une AMI personnalisée est créée à partir d’une instance existante.Le processus est simple :

    Lancer une instance à partir d’une AMI de base (ex. Amazon Linux).

    Installer les logiciels ou appliquer les configurations nécessaires (par exemple un serveur web ou une application métier).

    Créer une nouvelle AMI à partir de cette instance modifiée.

    L’AMI personnalisée conserve :

    le système d’exploitation de départ,

    les données préinstallées,

    et les configurations spécifiques apportées par l’utilisateur.

    Elle peut ensuite être :

    Réutilisée pour lancer d’autres instances avec exactement la même configuration,

    Partagée avec d’autres utilisateurs de manière contrôlée,

    ou rendue publique pour être disponible à une communauté plus large

    LAB : AMI personnalisée

    Dans ce scénario, nous prenons l’exemple d’un développeur web, Abdoulaye BASS.Lorsqu’il doit mettre un site en maintenance, il lance une instance EC2, installe le serveur httpd (Apache) et héberge une page de maintenance.L’objectif de ce LAB est de créer une méthode plus efficace qui lui permettra de gagner du temps : une AMI personnalisée contenant déjà Apache et sa page de maintenance.

    Pour commencer, il faut démarrer une instance EC2 à partir d’une Amazon Linux AMI.Une fois l’instance en cours d’exécution, nous procédons à l’installation du serveur Apache (httpd) afin de préparer l’hébergement de la page de maintenance.

    On sélectionne l’AMI Amazon Linux comme image de base pour lancer l’instance.

    On ajoute ensuite un script User Data qui installe automatiquement un serveur web (Apache) et déploie la page de maintenance lors du démarrage de l’instance.

    On crée un nouveau groupe de sécurité et on y ajoute une règle entrante pour autoriser le trafic HTTP (port 80), afin de rendre la page de maintenance accessible depuis un navigateur.

    On copie le nom DNS public de l’instance, puis on le colle dans la barre d’adresse d’un navigateur. La page de maintenance s’affiche, confirmant que le serveur web fonctionne correctement.

    On sélectionne l’instance EC2, puis dans le menu Actions → Images et modèles → Créer une image.Cette option permet de générer une AMI personnalisée directement à partir de l’instance en cours d’exécution.

    On attribue un nom et une description à l’image, par exemple : Maintenance-AMI02.

    Après avoir renseigné le nom et la description, on clique sur Créer l’image afin de lancer le processus de génération de l’AMI.

    Dans la section EC2 → AMI, on retrouve l’image Maintenance-AMI02 avec le statut En attente. AWS est en train de la créer.Une fois que son statut passe à Disponible, l’AMI pourra être utilisée pour lancer de nouvelles instances.

    On lance une nouvelle instance EC2 en sélectionnant l’AMI personnalisée créée précédemment.

    Dans la section Choix d’images, on clique sur Mes AMI, puis on sélectionne Maintenance-AMI02.

    On ajoute une règle entrante pour autoriser le trafic HTTP (port 80) depuis n’importe quelle adresse, puis on lance l’instance.

    Une fois l’instance démarrée, on copie le nom DNS public associé, par exemple :ec2-13-53-131-238.eu-north-1.compute.amazonaws.com.

    En accédant à ce DNS depuis un navigateur, la page de maintenance personnalisée s’affiche correctement.Dès lors, l’AMI personnalisée est disponible et peut être réutilisée à tout moment par BASS pour déployer rapidement une instance avec la page de maintenance déjà prête.

    4. Key Pair (Paires de clés)

    Les Key Pairs sont des paires de clés cryptographiques utilisées pour sécuriser l’accès aux instances EC2.Elles se composent de deux éléments :

    une clé publique, enregistrée automatiquement sur l’instance EC2 lors de son lancement,

    une clé privée, téléchargée par l’utilisateur et indispensable pour établir la connexion.

    Sans Key Pair, il est possible de lancer une instance EC2, mais il ne sera pas possible d’y accéder directement en SSH (Linux) ou en RDP (Windows).

    4.1 Fonctionnement avec EC2 Linux

    Lorsqu’une instance Linux est créée, AWS stocke automatiquement la clé publique dans le fichier d’authentification de la machine.La clé privée est téléchargée par l’utilisateur (au format .pem ou .ppk) et doit être conservée précieusement.Elle permet ensuite de se connecter via SSH à l’instance.Sans cette clé, l’accès est impossible, sauf en créant de nouvelles clés et en les réassociant manuellement.

    4.2 Fonctionnement avec EC2 Windows

    Pour une instance Windows, le mécanisme diffère.La clé publique est enregistrée sur l’instance, tandis que la clé privée permet de déchiffrer le mot de passe administrateur généré par AWS.Ce mot de passe est ensuite utilisé pour se connecter à l’instance via le protocole RDP (Remote Desktop Protocol).

    4.3 Sécurité et bonnes pratiques

    Conserver la clé privée dans un emplacement sécurisé et éviter de la partager.

    Ne jamais modifier ses permissions : sous Linux, la clé .pem doit être protégée avec la commande chmod 400.

    Préférer la création de Key Pairs distinctes par projet ou par utilisateur afin de mieux contrôler les accès.

    Supprimer ou révoquer les Key Pairs inutilisées pour limiter les risques de compromissi

    5. Adressage IP et Réseau

    Chaque instance EC2 est déployée dans une région AWS, au sein d’un VPC (Virtual Private Cloud).Le VPC est découpé en sous-réseaux (Subnets), et chaque instance doit être associée à l’un d’entre eux.

    5.1 Adresses privées et publiques

    Adresse IP privée : attribuée automatiquement par AWS, elle permet à l’instance de communiquer avec d’autres ressources du même VPC (par exemple au sein du même sous-réseau). Cette adresse reste inchangée, même si l’instance est arrêtée ou redémarrée.

    Adresse IP publique : optionnelle, elle permet à l’instance de communiquer directement avec Internet. Contrairement à l’adresse privée, l’adresse publique est réattribuée à chaque redémarrage de l’instance.

    5.2 Elastic IP

    Pour disposer d’une adresse publique fixe qui ne change pas, AWS propose les Elastic IP.Il s’agit d’adresses IP publiques statiques qui peuvent être associées à une instance EC2. Elles garantissent une connectivité constante depuis Internet, même en cas d’arrêt ou de redémarrage de l’instance.

    5.3 Intégration dans un VPC

    Les instances EC2 s’intègrent toujours dans un VPC.Le choix du VPC et du sous-réseau détermine :

    la zone de disponibilité dans laquelle l’instance sera déployée,

    les plages d’adresses IP disponibles,

    et les règles de communication applicables (via les groupes de sécurité et les ACL réseau).

    6. Groupes de sécurité

    6.1 Définition et rôle

    Un groupe de sécurité agit comme un pare-feu virtuel pour les instances EC2.Il contrôle le trafic entrant (inbound) et sortant (outbound) afin de protéger l’instance contre les accès non autorisés.On peut comparer cela à un vigile posté devant une maison : il dispose d’une liste qui précise qui est autorisé à entrer ou à sortir.

    Un groupe de sécurité peut être associé à une ou plusieurs instances EC2. Si plusieurs instances partagent les mêmes règles, il suffit de leur appliquer le même groupe de sécurité.

    6.2 Règles entrantes et sortantes

    Règles entrantes (Inbound) : définissent le trafic autorisé à entrer dans l’instance (par exemple, HTTP sur le port 80 ou SSH sur le port 22).

    Règles sortantes (Outbound) : définissent le trafic que l’instance est autorisée à émettre (par exemple, accès à Internet ou à d’autres instances du VPC).

    Les règles peuvent être restreintes selon plusieurs critères :

    l’adresse IP source ou destination,

    le protocole (TCP, UDP, ICMP),

    le port utilisé.

    ⚠️ Par défaut, tout trafic qui n’est pas explicitement autorisé est automatiquement bloqué (deny implicite).De plus, il est obligatoire d’associer au moins un groupe de sécurité à chaque instance EC2.

    6.3 Lab groupes de securité

    Pour ce test, nous allons réutiliser l’instance EC2 créée précédemment pour la page de maintenance.L’objectif est d’expérimenter différentes règles afin de comprendre leur effet sur le trafic réseau.

    Il est important de souligner que les modifications apportées à un groupe de sécurité s’appliquent immédiatement. Il n’est donc pas nécessaire de redémarrer ou de reconfigurer l’instance : les règles sont mises en œuvre à chaud.

    Le groupe de sécurité associé à cette instance est nommé launch-wizard11.Dans la console AWS, en allant dans EC2 → Groupes de sécurité, on peut visualiser et modifier ses règles.

    En cliquant sur l’ID du groupe de sécurité, on accède à la liste de ses règles.

    Règles entrantes (Inbound) :

    Une règle autorise le trafic HTTP sur le port 80 (TCP) depuis n’importe quelle adresse (0.0.0.0/0). Cela signifie que tout utilisateur sur Internet peut accéder à la page web hébergée sur l’instance.

    Une autre règle autorise les connexions SSH (port 22) depuis toutes les sources, permettant à quiconque de tenter une connexion à distance.

    En cliquant sur Modifier les règles, nous supprimons la règle entrante qui autorisait le trafic HTTP (port 80).Rappelons que l’instance EC2 héberge une page de maintenance via un serveur web : sans cette règle, la page ne pourra plus être accessible depuis Internet.

    Après avoir supprimé la règle HTTP, la page web devient inaccessible : le navigateur indique que le site est indisponible.Cela s’explique par le fonctionnement des groupes de sécurité, qui appliquent un principe de deny implicite : tout trafic non explicitement autorisé est automatiquement bloqué.

    Nous ajoutons de nouveau la règle HTTP sur le port 80, puis enregistrons les modifications.

    Après avoir rétabli la règle HTTP et actualisé la page dans le navigateur, l’accès à la page de maintenance est immédiatement rétabli.

    Par défaut sur les règles sortantes on voit qu’AWS autorise par défaut tous les trafics sortant sur tous les port vers n’importe quelles destinations donc on doit pouvoir joindre internet.

    A Partir de notre instance on voit qu’on parviens a joindre le servur DNS google.

    En exécutant la commande curl google.com depuis l’instance, la connexion aboutit correctement et renvoie un code HTTP 200, ce qui confirme que l’accès à Internet est bien fonctionnel.

    Nous supprimons la règle sortante, enregistrons les modifications, puis testons à nouveau la connectivité afin d’observer le résultat.

    Après suppression de la règle sortante, l’instance ne parvient plus à joindre le serveur Google : la connexion vers Internet est bloquée.

    La suppression de la règle sortante empêche toute communication : l’instance ne peut plus établir de connexion vers l’extérieur.

    En résumé, les groupes de sécurité permettent de contrôler efficacement le trafic entrant et sortant des instances EC2. Ils offrent une couche de protection essentielle et flexible. Bien que ce LAB illustre les règles de base, il est possible d’aller beaucoup plus loin en affinant les configurations pour répondre à des besoins de sécurité plus complexes.

    7. Volumes EBS

    7.1 Définition et caractéristiques

    Les volumes EBS (Elastic Block Store) sont des disques de stockage attachés aux instances EC2 via le réseau.Ils ne résident pas sur le même matériel physique que l’instance, mais sont reliés de façon transparente, comme s’il s’agissait d’un disque local.Les volumes EBS offrent une haute disponibilité, une résilience et peuvent être sauvegardés sous forme de snapshots pour assurer la persistance des données.

    7.2 Volume racine et volumes additionnels

    Volume racine (Root volume) : c’est le disque principal créé par défaut lors du lancement d’une instance EC2. Il contient le système d’exploitation et ne peut pas être détaché tant que l’instance existe.

    Volumes additionnels : il est possible d’attacher un ou plusieurs volumes EBS supplémentaires pour stocker des données, des applications ou des bases de données. Ces volumes peuvent être détachés et réattachés à tout moment, même si l’instance est arrêtée.

    8. Types d’instances

    8.1 Familles et générations

    Les types d’instances EC2 définissent les caractéristiques techniques mises à disposition : nombre de vCPU, mémoire, performance réseau et type de stockage.AWS regroupe ses instances en familles, chacune répondant à un cas d’usage spécifique :

    T : usage général, équilibre entre calcul, mémoire et réseau (idéal pour des applications légères ou de test).

    M : usage polyvalent, adapté aux applications d’entreprise.

    C : instances optimisées pour le calcul intensif.

    R : instances optimisées pour la mémoire (bases de données, cache).

    G : instances optimisées pour le GPU (IA, rendu graphique).

    I/D/H : instances optimisées pour le stockage (bases de données haute performance, big data).

    Chaque famille évolue au fil du temps avec plusieurs générations (ex. T2, T3, T4g), qui offrent de meilleures performances et un meilleur rapport prix/efficacité.

    8.2 Exemple de nomenclature

    La nomenclature des types d’instances suit une logique précise.Exemple : t2.micro

    t : famille de l’instance (usage général),

    2 : génération,

    micro : taille, indiquant le niveau de ressources (CPU, mémoire).

    Autre exemple : m6g.12xlarge

    m : famille (usage polyvalent),

    6 : génération,

    g : type de processeur (ici, Graviton basé sur ARM, a désigne AMD, absence de lettre = Intel),

    12xlarge : taille, correspondant à une configuration avec un grand nombre de vCPU et de mémoire.

    8.3 Choisir un type adapté aux besoins

    Le choix d’un type d’instance dépend directement du cas d’usage :

    Pour des tests ou environnements légers, une instance de type t2.micro ou t3.micro suffit (souvent éligible à l’offre gratuite).

    Pour une application en production nécessitant plus de puissance, il vaut mieux choisir une famille M ou C selon qu’on privilégie mémoire ou calcul.

    Pour des charges spécialisées (IA, bases de données lourdes, rendu graphique), il est préférable d’opter pour les familles R, G ou I/D/H.

    👉 AWS fournit une documentation officielle des types d’instances (AWS Instance Types) ainsi que des outils pratiques comme Vantage Instances qui permettent de comparer prix, caractéristiques et régions avec des filtres très utiles.

    LAB : Changement de type d’instance

    Dans certains cas, les besoins d’une application évoluent :

    Une augmentation de charge peut nécessiter plus de puissance (CPU, mémoire, réseau),

    À l’inverse, une baisse d’activité peut justifier le passage à une instance plus petite pour réduire les coûts.

    La solution consiste à modifier le type d’instance EC2 (par exemple passer d’un t2.micro à un t3.micro).

    ⚠️ Important : ce changement n’est pas possible à chaud. L’instance doit être arrêtée (Stop) avant de modifier sa configuration.De plus, ce type de manipulation ne peut pas être réalisé dans le cadre de ce LAB, car AWS facture l’utilisation du nouveau type d’instance.

    Étapes théoriques :

    Arrêter l’instance EC2.

    Dans la console AWS, aller sur Actions → Instance Settings → Modifier le type d’instance.

    Sélectionner le nouveau type (par ex. t3.micro).

    Valider la modification.

    Redémarrer l’instance, qui fonctionnera alors avec ses nouvelles ressources.

    9. Mise à l’échelle (Scaling)

    9.1 Mise à l’échelle verticale

    La mise à l’échelle verticale (ou vertical scaling) consiste à augmenter les ressources d’une instance existante.Par exemple, passer d’une base de données de 4 vCPU / 8 Go de RAM à une machine plus puissante.C’est une solution simple à mettre en œuvre, mais elle présente une limite : l’instance reste un point unique de défaillance (SPOF – Single Point of Failure). Si cette machine tombe en panne ou si la zone de disponibilité est affectée, tout le service est indisponible.

    9.2 Mise à l’échelle horizontale

    La mise à l’échelle horizontale (ou horizontal scaling) consiste à ajouter de nouvelles instances pour répartir la charge.Cette approche réduit les risques de SPOF, car la charge de travail est distribuée entre plusieurs serveurs.Par exemple, au lieu de renforcer une seule instance, on déploie deux instances t3.micro pour encaisser la charge.

    11. Elastic Load Balancer (ELB)

    11.1 Rôle et fonctionnement

    Un Elastic Load Balancer (ELB) est un service managé d’AWS qui assure la répartition automatique du trafic entre plusieurs instances EC2 ou autres cibles.Il est souvent utilisé en complément d’un groupe Auto Scaling, afin de distribuer les requêtes des utilisateurs vers les instances disponibles et en bonne santé.

    On peut l’imaginer comme un chef de projet : les clients envoient leurs requêtes à une seule adresse (le Load Balancer), et celui-ci se charge de les rediriger vers la bonne instance.Le Load Balancer ne gère pas directement les instances (ce rôle revient à Auto Scaling), mais il garantit que le trafic est correctement réparti.

    L’ELB permet donc :

    d’assurer la haute disponibilité,

    d’améliorer la tolérance aux pannes,

    et de faciliter la scalabilité des applications.

    11.2 Types de Load Balancer

    AWS propose plusieurs variantes de Load Balancer adaptées à différents besoins :

    Application Load Balancer (ALB) : fonctionne sur la couche 7 (Application) du modèle OSI.→ Principalement utilisé pour les sites web (HTTP/HTTPS).→ Permet un routage avancé basé sur l’URL, l’hôte ou les en-têtes.

    Network Load Balancer (NLB) : agit sur la couche 4 (Transport).→ Idéal pour des charges massives (TCP/UDP).→ Capable de traiter des millions de requêtes par seconde avec une faible latence.

    Gateway Load Balancer (GWLB) : fonctionne sur la couche 3 (Réseau).→ Utilisé pour intégrer des appliances virtuelles tierces (firewall, inspection réseau, monitoring).→ Fait office de passerelle tout en répartissant le trafic.

    Classic Load Balancer (CLB) : première génération de Load Balancer AWS.→ Aujourd’hui déprécié, il est recommandé de privilégier ALB ou NLB.

    11.3 Exemple pratique d’intégration avec Auto Scaling

    Prenons le cas d’un site de streaming hébergé sur 20 instances EC2.Sans Load Balancer, les internautes ne sauraient pas vers quelle instance envoyer leurs requêtes.Avec un ELB connecté à un groupe Auto Scaling, le trafic est distribué intelligemment entre les instances disponibles, garantissant performance et disponibilité.

    12. Architecture Multi-AZ et modèles de tarification EC2

    12.1 Architecture Multi-AZ (Availability Zone)

    Une architecture Multi-AZ consiste à déployer une application dans plusieurs zones de disponibilité d’une même région AWS.Concrètement, les instances EC2 sont réparties sur différentes zones, ce qui permet d’assurer une haute disponibilité.

    La haute disponibilité se définit comme la capacité d’un système à rester accessible et fiable presque en permanence, en minimisant les interruptions de service.Pour y parvenir, on met en place des redondances : plusieurs instances du serveur web, un groupe Auto Scaling pour garantir le bon nombre d’instances nécessaires, et un Load Balancer pour distribuer le trafic entre elles.

    12.2 Modèle On-Demand

    Le modèle On-Demand fonctionne selon le principe du pay-as-you-go : on ne paie que pour ce que l’on consomme, à l’heure ou à la seconde, sans engagement à long terme.

    Ce modèle est recommandé dans les cas suivants :

    environnements de test ou d’apprentissage,

    charges de travail à court terme (applications déployées quelques jours ou semaines),

    charges imprévisibles, lorsqu’on ne connaît pas à l’avance le nombre d’instances nécessaires.

    Ce modèle est très flexible, mais il n’est pas le plus économique sur le long terme.

    12.3 Savings Plans

    Les Savings Plans offrent une réduction allant jusqu’à 72 % par rapport au tarif On-Demand, en échange d’un engagement d’un ou trois ans sur une dépense horaire fixe.

    Trois options de paiement sont possibles :

    No Upfront : paiement mensuel, sans avance,

    Partial Upfront : paiement partiel à l’avance, puis le reste échelonné,

    All Upfront : paiement total dès le début du contrat.

    Ce modèle est recommandé pour les applications de production stables et les bases de données.

    12.4 Instances Spot

    Les Instances Spot exploitent les ressources EC2 inutilisées dans les datacenters AWS.Elles peuvent coûter jusqu’à 90 % moins cher que les instances On-Demand.

    Cependant, elles présentent une contrainte : AWS peut les récupérer à tout moment.En cas d’interruption, une notification est envoyée et l’utilisateur dispose de 2 minutes pour arrêter ou sauvegarder son travail.

    Elles sont particulièrement adaptées pour :

    les environnements de test,

    le Big Data,

    les pipelines CI/CD,

    et les applications tolérantes aux pannes.

    12.5 Dedicated Hosts

    Un Dedicated Host est un serveur physique entièrement dédié à un seul client dans un datacenter AWS.Il offre une visibilité complète sur les sockets et cœurs physiques, et permet d’utiliser des licences logicielles basées sur le matériel (par socket ou par cœur), comme Windows Server ou SQL Server.

    Ce modèle est recommandé pour :

    des besoins de conformité réglementaire,

    des charges de travail nécessitant un serveur physique isolé,

    des licences logicielles spécifiques
